# Persistence Architecture (Prompt 005)

The persistence layer implements the domain's repository contracts against
PostgreSQL using Prisma. It is **infrastructure**: it lives under
`src/infrastructure/persistence/prisma`, is server-only, and never leaks Prisma
types above itself. The domain and application layers are unchanged in shape;
persistence satisfies their existing contracts.

## Database choice

- **PostgreSQL** via **Prisma 6** (ADR-0008). No ORM abstraction is layered over
  Prisma — the domain repository contracts are already the abstraction boundary.
- Local/CI without Docker: integration tests run against a **real** PostgreSQL 17
  started in userspace by `embedded-postgres` (not SQLite, not an emulation).

## Layer boundaries

```
src/infrastructure/persistence/prisma/
  client.ts            # server-only Prisma client lifecycle
  errors.ts            # Prisma error → safe PersistenceError translation
  mappers/             # explicit row ↔ domain mappers (one per aggregate)
  repositories/        # PrismaXRepository adapters implementing domain contracts
  index.ts             # public surface (adapters, client, errors)
```

ESLint forbids `src/domain` and `src/application` from importing `@/infrastructure`
or `@prisma/client`, and forbids infrastructure from importing `@/application`,
`@/ui`, `@/app`, React, or Next. Persistence is wired at the composition root
(delivery layer, later), never imported by the layers it serves.

## Schema overview & table-to-aggregate mapping

| Aggregate | Root table | Owned child table(s) | Notes |
|-----------|-----------|----------------------|-------|
| Project | `projects` | — | `owner_id` indexed for owner lists |
| ContentItem | `content_items` | — | `slug` unique; `version` (domain revision) |
| Rubric | `rubrics` | `rubric_criteria` (cascade) | `position` preserves order |
| Evaluation | `evaluations` | `evaluation_scores` (cascade) | composite PK `(evaluation_id, criterion_id)` |
| Decision | `decisions` | `decision_options` (cascade) | advisory flattened to nullable columns |

Identifiers are the domain's branded strings (`String` PKs, never autoincrement),
generated by the application and validated by each mapper on read. `OwnerId` is
stored as an **external identity reference** — there is no users table
(authentication is deferred; no fake table was added to satisfy a foreign key).

## Constraints (database-enforced integrity)

- Primary keys on every table; composite PKs enforce unique aggregate-owned child
  ids (no duplicate criterion per evaluation; no duplicate option id per decision).
- Foreign keys: `rubric_criteria`, `evaluation_scores`, `decision_options` →
  their roots with **ON DELETE CASCADE** (children have no independent meaning);
  `evaluations`/`decisions` → `projects`/`rubrics` with **ON DELETE RESTRICT**
  (history is preserved, never cascade-deleted).
- Unique: `content_items.slug`, `rubrics.slug`.
- CHECK constraints (hand-authored migration): `rubric_criteria.weight > 0`;
  `evaluation_scores.score BETWEEN 1 AND 10`; `content_items.version >= 1`;
  `decisions.advisory_confidence` in `[0,1]` or null.

## Indexes and query justification

| Index | Query it supports | Why the PK is insufficient |
|-------|-------------------|----------------------------|
| `projects(owner_id)` | `listByOwner` | PK is `id`, not owner |
| `content_items(slug)` unique | `findBySlug` / `existsBySlug` + uniqueness | different column from PK |
| `rubrics(slug)` unique | rubric slug uniqueness | — |
| `rubric_criteria(rubric_id)` | load a rubric's criteria | criteria fetched by parent |
| `evaluations(project_id)` | `listByProject` | PK is `id` |
| `decisions(project_id)` | `listByProject` | PK is `id` |

No speculative indexes were added. `findById` uses the primary key.

## Prisma client lifecycle

`client.ts` (server-only) exposes `createPrismaClient()` and a `prisma` singleton
cached on `globalThis` in non-production to survive hot reload without leaking
connections. Query logging is disabled (only `warn`/`error`) so query parameters —
which may contain private content — are never logged. **Adapters receive a client
by constructor injection**, so tests inject a test-database client and the
production composition root injects the singleton. The singleton assumes a
long-lived server process; serverless deployments must revisit connection handling.

## Repository adapters & mapper strategy

Five adapters (`PrismaProjectRepository`, `PrismaContentRepository`,
`PrismaRubricRepository`, `PrismaEvaluationRepository`, `PrismaDecisionRepository`)
implement the domain contracts exactly. `AiRecommender` is **not** a persistence
repository and is not implemented here.

Mappers are **explicit** (no reflection, no auto-mapper), one module per aggregate,
each with `toDomain` (validates ids/values, rejects corrupt data via
`PersistenceMappingError`) and `to*Fields`/`to*Rows` for writes. Application views
are never reused as persistence models. Enum values map directly (the Prisma enum
and domain union are the same string literals; divergence is a compile error).

## Save semantics (per repository)

`save` was replaced by **intent-revealing operations** so persistence never
silently guesses create-vs-update (upsert would let a mutation on a missing row
insert, or a create with a colliding id overwrite):

| Repository | Operations | Missing row on update | Existing id on insert |
|-----------|-----------|-----------------------|-----------------------|
| Project | `insert`, `update` | `NotFoundError` (never inserts) | `ConflictError` (never overwrites) |
| Content | `insert`, `update` | `NotFoundError` | `ConflictError` (slug also unique) |
| Rubric | `insert` (append-only) | n/a | `ConflictError` |
| Evaluation | `insert` (append-only) | n/a | `ConflictError` |
| Decision | `insert`, `update` | `NotFoundError` | `ConflictError` |

- **`insert`** = create only. A duplicate primary key is rejected (`ConflictError`).
- **`update`** = update only, via **compare-and-swap on `lockVersion`** (see
  concurrency). `updateMany WHERE id = ? AND lock_version = ?` affecting zero rows
  is resolved to `OptimisticConcurrencyError` (row exists, version moved) or
  `NotFoundError` (row absent) — never an insert.
- Rubric and Evaluation are **append-only** (no update use case); they expose only
  `insert`.
- Creation use cases call `insert`; lifecycle/mutation use cases call `update`.

## Missing-record and duplicate-id behavior

- Updating a non-existent aggregate throws `NotFoundError` and writes nothing
  (verified for Project, Content, Decision).
- Inserting an existing id throws `ConflictError` and leaves the original
  unchanged (verified; append-only Rubric/Evaluation reject duplicate insertion).
- A failed write (conflict, stale version, FK violation) leaves the prior record
  and any aggregate children unchanged (transactions roll back).

## Transaction boundaries

- Rubric, Evaluation, Decision **inserts** are atomic (root + children in one
  `prisma.$transaction`). A mid-transaction failure rolls back with no partial
  write (verified: a foreign-key violation on an evaluation leaves no rows).
- Decision **update** touches only the root row (options are immutable after
  proposal), so it needs no transaction; Project/Content updates are single-row.
- No unit-of-work framework and no cross-repository transaction exist. No current
  workflow saves multiple aggregates atomically; if one arises it must be called
  out explicitly (tracked in the backlog).

## Uniqueness behavior

`content_items.slug` uniqueness is **database-authoritative**. `createContentItem`
pre-checks for a friendly error but does not rely on it: a race that passes the
pre-check hits the unique index, surfaces as a neutral `ConflictError`
(`@/lib/errors`), and is mapped to the same `SlugAlreadyExistsError`. (ContentItem
ids are application-generated and never collide, so the only realistic content
conflict is the slug.)

## Ownership query behavior

`listByOwner` filters by `owner_id` **in the database** (never load-then-filter).
Evaluation/decision ownership is enforced in the application by loading the owning
project; the FKs guarantee the project/rubric references are valid.

## Concurrency strategy (per aggregate)

**Optimistic concurrency is implemented** for the mutable aggregates, ahead of the
Prompt 006 delivery layer that will introduce concurrent requests. Classification:

| Aggregate | Classification | Concurrency handling |
|-----------|----------------|----------------------|
| Project | mutable, lost-update-sensitive | integer `lockVersion` + CAS on update |
| Content | mutable, lost-update-sensitive | integer `lockVersion` + CAS on update |
| Decision | mutable, lost-update-sensitive | integer `lockVersion` + CAS on update |
| Rubric | append-only (no update use case) | none (insert-only) |
| Evaluation | append-only (immutable once recorded) | none (insert-only) |

Mechanism: a provider-neutral integer `lockVersion` on each mutable aggregate root,
persisted, loaded with the aggregate, and checked in the update predicate
(`WHERE id = ? AND lock_version = ?`) while atomically incrementing it. A mismatch
yields a typed `OptimisticConcurrencyError` (`@/lib/errors`, distinguishable from a
generic failure). `lockVersion` is **distinct from** the Content domain `version`
(revision count) and is **not exposed in application views** (no caller needs it
yet); command inputs do not require an expected version, because the current
load-modify-update use cases carry it on the loaded aggregate. This guarantees a
stale write cannot silently overwrite newer state — including that a **duplicate
human finalization cannot overwrite a valid decision** (verified in integration
tests). Rubric/Evaluation get no concurrency machinery (no evidence of need).

## Deletion & retention

Archive states, not hard deletes — no `delete` methods are exposed. Evaluations and
decisions are never cascade-deleted (their project/rubric FKs are `RESTRICT`).
Only genuinely-owned child rows cascade with their root.

## Error translation

`translatePrismaError` maps structured Prisma error codes (never message strings).
Outcomes the **application** recognizes become **neutral shared-kernel errors**
(`@/lib/errors`, so no layer imports another's error classes): `P2002` →
`ConflictError`, `P2025` → `NotFoundError`. Cases the application never inspects by
type stay infrastructure-owned: `P2003` → `ForeignKeyConstraintError`,
init/panic/unknown → `DatabaseUnavailableError`, corrupt persisted data →
`PersistenceMappingError`. Optimistic-concurrency conflicts are raised directly by
adapters as `OptimisticConcurrencyError` (`@/lib/errors`). Raw Prisma errors, SQL,
and credentials never escape infrastructure; causes are retained internally and
omitted from `toJSON`. The application's `attempt()`/`attemptUpdate()` translate
thrown errors into `RepositoryError` (or a passed-through `OptimisticConcurrencyError`)
for use-case results.

## Migration workflow

- Canonical: `prisma migrate deploy` (never `db push`). Migrations live in
  `prisma/migrations/` and are all part of normal Prisma migration history:
  1. `20260719050215_init_persistence` — tables, enums, constraints, indexes, FKs.
  2. `20260719050216_add_check_constraints` — hand-authored CHECK constraints.
  3. `20260719050217_add_optimistic_concurrency` — the `lock_version` columns.
- `migrate deploy` applies all three **in order** on an empty database, and
  `migrate status` reports "up to date" (verified every integration run, which
  starts a fresh database).
- **Reconciliation for constraints Prisma cannot express (documented, not an
  ad-hoc exception):** CHECK constraints are not representable in `schema.prisma`,
  so they live only in a migration. The canonical workflow is therefore
  **schema edits + `migrate deploy`**, never `migrate dev` (whose shadow-database
  diff would report the CHECK constraints as drift). Anyone changing the schema
  hand-authors a migration and applies it with `migrate deploy`; a developer
  following this documented workflow neither loses nor duplicates the constraints.
- Prisma does not generate down migrations; operational rollback is by restoring
  from backup (see backlog: migration rehearsal automation).

## Environment requirements

- `DATABASE_URL` (server-only, validated in `src/lib/env/env.server`).
- Integration tests use a fixed test URL (`localhost:54329/stroman_test`) served by
  embedded PostgreSQL; they never target development or production.

## Integration-test strategy

`npm run test:integration` (config `vitest.integration.config.ts`, node
environment) starts one real PostgreSQL via `test/db/global-setup.ts`, applies
migrations with `migrate deploy`, and runs `*.integration.test.ts` sequentially
(shared server; truncate between tests). If the database cannot start, setup
**throws** — persistence tests are never silently skipped. The fast unit suite
(`npm test`) excludes integration tests and needs no database.

## Deployment considerations

Long-lived server process assumed. For serverless, review Prisma connection
management and pooling before use (backlog). Production must run `migrate deploy`
as a release step and must never point migration commands at a test database.

## Explicit exclusions

No API/UI/server actions, authentication, users/orgs/RBAC tables, AI provider
adapters, file storage, queues, caching, or analytics. `AiRecommender` is not a
persistence repository.

## Known limitations

- Optimistic concurrency **is implemented** for mutable aggregates; append-only
  aggregates deliberately have none.
- No users table (`OwnerId` is an external reference until auth exists).
- No cross-repository transaction abstraction (no workflow needs one yet).
- CHECK-constraint migrations are invisible to `migrate dev` drift detection; the
  project uses `migrate deploy` as the canonical workflow (see Migration workflow).
